## 🏗️ Project Structure

### Step 1 — Database Schema (DDL)
- Designed a fully normalized relational database with 6 tables
- Implemented Primary Keys, Foreign Keys, and CHECK constraints
- Used PostgreSQL-specific syntax — TIMESTAMP, BOOLEAN, 
  GENERATED BY DEFAULT AS IDENTITY, DECIMAL

### Step 2 — Data Loading
- Bulk loaded 1.7M+ records using PostgreSQL COPY command
- Managed correct load order to respect Foreign Key dependencies
- Synced identity sequences after load
- Verified row counts across all 6 tables

### Step 3 — Complex JOINs & Subqueries
8 business-driven queries answering real analytical questions:
- Product profitability and margin analysis
- Monthly revenue trends
- Refund rate by product
- Cross-sell pattern analysis
- Top traffic sources by revenue
- Conversion rate by marketing channel
- Repeat vs new customer revenue
- Clean vs refunded order breakdown

### Step 4 — Window Functions & CTEs
8 advanced analytical queries using:
- LAG() — Month over month revenue growth
- RANK() — Product revenue ranking per year
- NTILE() — Customer segmentation by spending tier
- SUM() OVER() — Running total revenue
- AVG() OVER() — Product performance vs average
- Chained CTEs — Peak sales days per month

### Step 5 — Stored Procedures & Triggers
4 reusable stored procedures with parameterized inputs:
- get_product_sales_summary(product_name)
- get_monthly_revenue_report(year)
- get_refund_analysis(start_date, end_date)
- get_traffic_performance(start_date, end_date)

3 data quality triggers:
- Auto audit log for every new refund inserted
- Validation to block negative refund amounts
- Validation to block refunds exceeding original item price

![Stored Procedure — Refund Analysis by Date Range](Screenshot%202026-05-28%20011007.png)

### Step 6 — Performance Optimization & Indexing
- Created 21 strategic indexes across 5 tables
- Used simple and composite indexes on high-traffic columns
- Applied EXPLAIN ANALYZE before and after indexing
- Documented when NOT to use indexes — low cardinality, 
  small tables, excessive DML operations
